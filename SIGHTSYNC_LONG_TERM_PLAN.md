# SightSync 长期开发计划

本文档是后续跨对话执行 SightSync 长期开发的项目顶部入口。每个新对话开始执行前，应先读取 `AGENTS.md`、`纲领.md`、V2 规格文档，以及本文档。
新对话执行本计划时，应先使用 `using-superpowers` skill：`D:\codex\.codex\skills\using-superpowers\SKILL.md`。

## 执行原则

- 不一次性做完整路线。每次只执行一个小片，完成后运行对应自动化测试和 Android 模拟器或真机验证。
- 每个 Phase 内的小片可以连续推进，但必须在整个 Phase 完成后单独做一次阶段验收。
- 阶段验收通过前，不进入下一个 Phase。
- 每个小片默认使用 TDD：先写失败测试，再实现，再跑目标测试和必要回归。
- 每次 App 层代码修改后，按项目约束在本机 Android 虚拟机或真机验证；如果设备不可用，必须记录阻塞原因。
- 高危行为在当前长期计划内先不执行，包括支付、转账、输入密码、输入验证码、银行卡、身份证敏感提交、删除不可恢复内容。
- 采用个人云代理作为长期独立运行方案：手机只保存代理地址和 App token，真实 AI API key 保存在云端代理。
- `D:\文档\SightSync改进建议.md` 作为参考来源，不整包采纳；只在对应小片需要时评估其中建议。

## Phase 1：稳定性和安全底座

目标：先让现有单步能力更稳、更安全，不扩展新动作能力，不做独立云代理，不做多步 Agent。

小片：

1. 确认/取消词、脱敏、节点 bounds 校验。
2. TTS 回调并发安全。
3. HTTP 错误类型化、超时细分、一次有限重试。
4. TTS 初始化失败兜底和可见状态反馈。
5. 上下文风控：支付、密码、验证码等页面先拒绝执行。
6. Phase 1 阶段验收和验收记录整理。

Phase 1 验收门槛：

- `.\gradlew.bat :app:testDebugUnitTest` 通过。
- `cd backend; npm test` 通过；如果 shell 中存在真实 provider key，应清空测试进程中的 `QWEN_API_KEY`、`DASHSCOPE_API_KEY`、`AI_API_KEY` 后重跑。
- `.\gradlew.bat :app:assembleDebug` 通过。
- Android 虚拟机或真机安装 debug APK 后，无障碍服务可启用，连续聆听通知和悬浮窗可见，最近 logcat 无 SightSync 崩溃。
- 人工或半自动验收覆盖：停止/取消语义、TTS 被打断、网络错误反馈、敏感字段脱敏、高风险页面拒绝执行。

## Phase 2：脱离电脑运行

目标：手机脱离电脑，通过个人云代理完成 ASR、屏幕理解和动作规划。

小片：

1. App 内“AI 服务连接配置”数据结构和本地保存。
2. 配置 UI：代理地址、App token、连接测试、当前连接状态。
3. 后端代理健康检查、鉴权、错误归一化。
4. App 使用动态代理配置，替代固定 `BuildConfig.AI_PROXY_BASE_URL` 作为唯一入口。
5. Phase 2 阶段验收：真机断开电脑后完整跑通语音链路。

Phase 2 验收门槛：

- App 首次安装后可以配置代理地址和 App token。
- 连接测试能区分成功、鉴权失败、代理不可达、provider 不可用。
- 真机不连接电脑、不使用 `adb reverse` 时，能完成至少一次语音转写、读屏请求和低风险动作。
- App 内不保存第三方 AI provider 的真实 API key。
- 后端日志不得输出用户 token、真实 AI key、完整截图 base64 或未脱敏敏感字段。

## Phase 3：屏幕理解 V2

目标：从“列出节点文字”升级为“理解复杂页面并清楚叙述”。

小片：

1. 屏幕摘要协议和后端 prompt 调整。
2. 节点树提取增强：层级、区域、可操作项、滚动容器、输入框上下文。
3. 读屏模式：简短读屏、详细读屏、可操作项说明。
4. 截图辅助理解策略和隐私控制。
5. Phase 3 阶段验收：复杂页面读屏质量验收集。

Phase 3 验收门槛：

- 常见设置页、浏览器网页、聊天列表、输入页面能给出结构化叙述，而不是只朗读前几个控件。
- 读屏结果包含页面主题、主要内容、可操作项和风险提示。
- 节点树足够时不附带截图；节点不足时才按策略附带截图。
- 敏感字段仍在本地优先脱敏。
- 复杂页面理解失败时，助手应说明无法确定，而不是编造页面内容。

## Phase 4：低风险受限多步 Agent

目标：实现浏览器搜索、App 内导航、微信发送草稿等低风险多步任务。仍不做自由自主代理。

小片：

1. 多步计划协议和 Android 端校验器。
2. 内部等待和页面稳定校验。
3. 浏览器搜索任务：打开浏览器、定位搜索/地址栏、输入内容、提交搜索。
4. App 内导航和候选消歧：找不到或候选不唯一时追问，不猜测执行。
5. 微信发送草稿流程：找到联系人、填写消息、最终发送前确认；当前阶段不自动发送高风险或不确定内容。
6. Phase 4 阶段验收：成功、页面变化中断、高风险确认、用户主动停止四类场景。

Phase 4 验收门槛：

- 多步任务只由用户明确语音发起。
- 每一步都是白名单动作或已批准的内部等待/校验步骤。
- 每一步执行后重新采集或校验页面状态；页面变化、目标丢失、超时、连续失败时停止后续步骤并播报原因。
- 用户点击浮窗、通知停止入口或说停止类指令后，不再执行后续步骤。
- 浏览器搜索能在测试设备上完成；微信流程必须停在最终确认，不自动越过确认。

## Phase 5：真实使用打磨

目标：把前四个 Phase 的能力变成可持续验收和日常使用的体验。

小片：

1. 高频 App 验收清单：微信、浏览器、设置、电话、短信等。
2. 失败日志和诊断工具：聚焦语音链路、屏幕理解、动作执行和代理错误。
3. 语音话术和交互细节打磨：少打扰、可中断、失败原因清晰。
4. Phase 5 阶段验收：回归验收自动化或半自动化。

Phase 5 验收门槛：

- 形成可重复的真机验收清单和日志归档方式。
- 常见失败能从日志判断发生在 ASR、AI 规划、屏幕采集、动作执行还是权限层。
- 高频 App 的核心低风险场景有明确通过/失败记录。
- 语音交互不会反复播报无效提示，不会在连续聆听中频繁打扰用户。

## 跨对话接续方式

新对话开始时按以下顺序执行：

1. 读取 `AGENTS.md`、`纲领.md`、`docs/superpowers/specs/2026-05-18-v2-speech-stability-design.md`、`docs/superpowers/specs/2026-05-28-v2-boundary-update-design.md` 和本文档。
2. 使用 `using-superpowers` skill：`D:\codex\.codex\skills\using-superpowers\SKILL.md`，再按任务性质使用必要的其他 skill。
3. 查看 `git status --short`，识别已有未提交改动，不回滚无关改动。
4. 在本文档中找到当前 Phase 和下一个未完成小片。
5. 只执行该小片；如果需要扩大范围，先说明原因。
6. 小片完成后报告：改动摘要、测试命令、Android 验证结果、剩余风险、下一个小片。
7. 整个 Phase 的所有小片完成后，先做 Phase 阶段验收并记录结果，再进入下一 Phase。
