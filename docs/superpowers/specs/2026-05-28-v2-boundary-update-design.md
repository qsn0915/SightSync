# SightSync V2 边界更新设计

## 目标

本规格更新 SightSync V2 的产品与工程边界。用户已确认：继续保留隐私、安全和可见控制底线，同时放宽部分旧的阶段边界，使 SightSync 可以从单步语音动作演进到受限多步 Agent 任务。

本规格不采纳 `D:\文档\SightSync改进建议.md` 中的具体代码修改建议；那些建议暂时搁置，后续由用户明确要求时再单独评估和实现。

## 不可突破的底线

以下边界继续作为 V2 安全底线，不能因为多步 Agent 能力而放松：

- 继续基于现有 SightSync Android App 演进，不新建替代 App 或替代 Android 模块。
- 只支持用户显式开启后的连续聆听；不做隐形热词唤醒、隐藏后台录音或绕过 Android 前台服务和通知要求的麦克风使用。
- 连续聆听运行时必须保持可见通知和悬浮窗状态，并提供浮窗、通知和语音停止入口。
- Android App 内不硬编码云端 AI API key，不自行选择或切换 AI provider/model；AI 服务统一通过后端代理接入。
- AI 不得返回任意脚本、shell、代码或自由命令；Android 端只执行已批准的结构化白名单动作。
- 高风险动作必须二次确认；支付、转账、删除、发送、提交、注销、退出登录和类似不可逆或对外产生影响的操作不能一次语音直接执行。
- 屏幕和应用数据遵循最小化原则：优先上传无障碍节点树，只有节点不足时才按既有策略附带截图；已安装应用列表默认只在本地使用，不上传后端。

## 更新后的阶段边界

### 受限多步 Agent

SightSync 可以从“单步白名单动作”扩展为“受限多步白名单任务”。多步任务必须满足：

- 任务由用户明确发起，不能由 App 自主长期探索或主动执行。
- 每一步都必须是白名单动作或本地内部等待/校验步骤。
- 每一步执行前后都必须能校验当前页面状态；校验失败时停止后续步骤并语音说明。
- 任意高风险步骤触发二次确认；确认前不得继续执行后续动作。
- 用户点击浮窗、通知停止入口或说停止类指令时，必须停止后续未执行步骤；已执行动作不回滚，但要明确反馈当前状态。
- 多步任务必须限制最大步数、最大总耗时和连续失败次数，防止陷入循环。

受限多步 Agent 不是完整自主代理。它不能“自己想办法完成任何目标”，不能在没有用户明确指令时跨 App 探索，也不能绕过动作白名单、安全确认或语音状态机。

### 动作白名单扩展原则

动作白名单可以迭代扩展，但每个新增动作必须单独进入规格或实现计划，并说明安全级别、校验方式和验收场景。

优先考虑的扩展方向：

- 内部等待机制：允许动作执行器在多个动作之间等待 UI 稳定，或新增受控 `WAIT` 动作。
- 页面状态校验：允许多步任务在继续前确认目标包名、节点文本、节点位置或页面关键字仍符合预期。
- 更强的本地解析：例如应用名别名、本地候选消歧和安全上下文判断。

暂不优先扩展：

- 任意坐标点击、复杂手势轨迹、剪贴板读取、通知批量操作、文件系统操作和跨 App 分享。
- 这些能力只有在明确用户场景、风险分类、确认策略和真机验收标准齐备后才能进入实现。

### 上下文风控

风控边界从“用户语音和目标文本关键词”扩展为“上下文风控”。后续实现应逐步纳入：

- 当前包名或应用类别：支付、银行、电商、聊天、邮箱、文件管理等应用中的提交类动作风险更高。
- 当前页面内容：页面包含支付、付款、转账、订单、删除、发送、提交、验证码、银行卡等关键词时，提高点击和输入动作风险等级。
- 动作序列上下文：多步任务中，如果前序步骤进入高风险页面，后续“确定”“下一步”“完成”等泛化按钮不能视为普通点击。

上下文风控可以产生误报，但不能为了减少打扰而漏掉明显高风险场景。误报通过更清晰的确认话术和取消语义缓解。

### 语音优先，视觉兜底

SightSync 仍以盲人用户的语音体验为主。所有关键状态和结果应优先通过 TTS 播报。

允许增加非语音兜底反馈：

- 当系统 TTS 初始化失败或不可用时，可以通过通知、悬浮窗状态和 Toast 显示错误。
- 视觉兜底不能替代语音主流程，也不能让核心功能依赖用户看屏幕。
- 即使 TTS 不可用，语音状态机仍必须避免录音、朗读、请求和动作执行并发错乱。

### 应用覆盖范围

SightSync 不再只停留在系统设置、浏览器、短信/聊天列表的 V1 验收范围，但新增 App 覆盖必须按场景逐步批准。

优先覆盖顺序：

- 系统设置、浏览器、电话、短信、主流聊天 App 的基础读取和基础动作。
- 用户明确指定的常用 App 的打开、读取、点击、输入和返回。
- 支付、银行、电商、文件删除等高风险 App 先做读取和低风险导航，再逐步放开确认后的动作执行。

不得口头声明“支持所有 App”。每个新增重点 App 或任务类型都需要有可重复的验收步骤。

## 多步任务执行模型

后续实现多步 Agent 时，应采用“计划-执行-校验”的小步循环：

```text
User utterance
  -> classify local command / remote planning
  -> produce bounded action plan
  -> speak intended step or summary
  -> execute one step
  -> wait for UI stability if needed
  -> collect/validate current state
  -> continue, ask confirmation, ask clarification, or stop
```

计划可以来自本地规则或后端 AI，但 Android 端必须重新校验协议、风险和当前页面状态。后端 AI 给出的计划不是执行授权。

## 与既有规格的关系

- 本规格优先于 V1 `纲领.md` 中“不做复杂自动化流程”的阶段性限制，但不突破 V1 的动作白名单、安全确认和隐私底线。
- 本规格不替代 `2026-05-18-v2-speech-stability-design.md`。多步任务必须继续使用半双工语音状态机和明确取消语义。
- 本规格扩展 `2026-05-25-v2-local-agent-execution-design.md`。本地打开应用仍是基础 Agent 能力的一部分，多步任务是在它之后的受限演进。

## 验收标准

后续任何多步 Agent 实现至少满足：

- 多步任务只由用户显式语音发起。
- 每一步动作均属于白名单或明确批准的内部控制步骤。
- 页面变化、目标节点丢失、包名变化、超时或连续失败时停止后续步骤并播报原因。
- 高风险上下文中的确认、提交、发送、付款、删除类动作必须二次确认。
- 用户在朗读、录音、请求、等待、动作执行任一阶段停止时，不再执行后续步骤。
- 单元测试覆盖计划边界、动作校验、上下文风控、停止语义和失败中断。
- 本机 Android 虚拟机或真机验收覆盖至少一个成功多步低风险任务、一个页面变化中断任务、一个高风险确认任务和一个用户主动停止任务。

## 明确不做

- 不做隐藏监听、隐形热词唤醒或未授权后台录音。
- 不做自由脚本执行、shell 执行、任意代码执行或非白名单动作执行。
- 不让后端 AI 直接决定安全策略；Android 端必须保留最终协议校验和风险判断。
- 不上传全量已安装应用列表，除非用户之后单独批准新的隐私规格。
- 不一次性实现完整自主代理；只按可验收场景逐步扩展受限多步任务。
