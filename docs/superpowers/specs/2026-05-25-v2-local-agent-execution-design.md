# SightSync V2 本地 Agent 执行设计

## 目标

SightSync V2 在现有 Android App 和后端代理基础上，增加基础 Agent 执行层：用户显式开启连续聆听后，可以通过语音下达明确指令，SightSync 将其转换为白名单动作并执行。首要场景是“帮我打开浏览器”“打开设置”“启动微信”等打开已安装 App 的命令，同时继续收紧语音状态、取消语义和动态录音稳定性。

本阶段不做完整自主多步代理、不做隐形热词唤醒、不上传已安装应用列表、不新增 AI provider/model、不在 Android App 内硬编码云端 AI API key。

## 本地打开应用能力

Android 端新增本地应用目录和命令解析器。目录只读取可启动应用的显示名称和包名，并在手机本地归一化匹配，不发送给后端代理。用户说“打开/启动/帮我打开 X”时，Session 在采集屏幕和调用后端 AI 前先尝试本地解析：

- 唯一匹配：生成现有 `OPEN_APP` 动作并走协议校验、风险确认和 `ActionRunner`。
- 多个匹配：语音追问用户说得更具体，不执行动作。
- 无匹配：回落到现有屏幕采集和后端 AI 规划。
- “浏览器”优先使用系统默认 Web 浏览器；没有默认浏览器时按本地应用名/别名匹配。

为适配 Android 11+ package visibility，manifest 只声明 launcher intent 的 `<queries>`，不申请 `QUERY_ALL_PACKAGES`。

## 语音稳定性

现有半双工 `VoiceTurnCoordinator` 继续作为语音时序边界。新增本地打开应用路径也必须在同一 turn 内执行，不能绕过可等待 TTS：

- 打开应用前先朗读将要执行的动作。
- 执行动作后等待执行反馈或错误提示朗读完成，再进入下一轮连续聆听。
- 高风险确认待处理中说“停止/取消/退出”等停止语义时，取消高风险操作并结束连续聆听。
- 扩展停止词，但停止指令不调用屏幕采集或后端 AI。

录音层将 `ShortAudioRecorder` 内部从 `MediaRecorder.maxAmplitude` 轮询改为 `AudioRecord` PCM 录音，并输出 WAV。通过帧能量检测静音尾段，保持约 800ms 最短录音、900ms 静音结束阈值和 8s 最长录音，不引入流式 ASR。

## 安全与隐私

所有执行仍受既有 V1/V2 底线约束：只执行白名单动作；高风险动作需要二次语音确认；页面点击、输入、滚动继续使用现有节点校验；应用列表只在本地使用。隐私说明补充“会在本地读取可启动应用名称以支持打开应用，不上传应用列表”。

## 验收标准

- 说“帮我打开浏览器”能打开默认浏览器或唯一匹配的浏览器 App。
- 说“打开设置”能打开系统设置或本机可启动设置入口。
- 多个候选应用时只追问，不猜测执行。
- “停止聆听/停止/取消/退出”等停止词在连续聆听和高风险确认中行为一致。
- 短命令不等待固定 4 秒，慢速长命令不在 4 秒截断，TTS 播报不会被下一轮录音捕获。
- Android 单元测试、后端测试、debug APK 构建通过；本机模拟器完成打开应用和语音稳定人工验收，或明确记录环境阻塞。
