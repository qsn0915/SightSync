# SightSync V2 语音稳定性设计

## 目标

SightSync V2 的第一阶段先解决 V1 的语音混乱问题，范围限定在现有 SightSync Android App 和已有后端代理内。完成后，助手应避免把自己的 TTS 朗读录进去，避免固定 4 秒录音造成用户话没说完被截断或说完后空等，并让连续聆听、取消、朗读、录音、AI 请求之间有明确状态边界。

本阶段不扩展手机控制能力，不新增 AI provider，不在 Android App 内保存云端 AI API key，不引入全天候热词唤醒或隐形后台常驻录音。

## 问题定义

V1 现有语音链路存在三个直接风险：

- `TtsOutputController.speak()` 只发起朗读，不等待朗读完成。连续聆听循环可能在 TTS 仍播放时开始下一轮录音，导致助手听见自己的声音。
- `ShortAudioRecorder` 固定录制 4 秒。用户说得慢或句子长时会被截断，用户说得短时又会产生不必要等待。
- `AssistantSessionManager` 同时管理连续聆听、一次请求、TTS、录音取消、AI 请求和动作执行。状态转换不够显式，点击浮窗或说停止指令时容易出现重复提示、交叉取消或下一轮提前开始。

## 推荐方案

采用半双工语音状态机：任何时刻只允许一个语音阶段占用麦克风或扬声器。TTS 播放时不录音，录音时不播放 TTS；一轮语音命令按固定顺序执行：

```text
Idle -> SpeakingPrompt -> Listening -> Transcribing -> Thinking -> SpeakingResult -> Acting -> NextListening/Idle
```

连续聆听不是独立执行路径，而是重复调度同一个 turn。每一轮必须等上一轮的 TTS、AI 请求和动作执行收束后，才进入下一轮录音。

## 组件边界

### VoiceTurnCoordinator

新增 `VoiceTurnCoordinator` 或等价小组件，负责单轮语音的时序控制：

- 播放提示语，并等待提示语结束后再启动录音。
- 调用录音器获取用户语音。
- 调用 ASR 代理转写。
- 将识别结果交回 `AssistantSessionManager` 处理屏幕采集、AI 规划和动作执行。
- 为取消流程提供单一入口，避免 TTS、录音和请求各自随意停止。

该组件不理解屏幕节点、不调用 AI 规划、不执行无障碍动作。

### AwaitableSpeechOutput

将当前 fire-and-forget 的 TTS 输出升级为可等待接口：

- `speakAndAwait(text)`：开始朗读并挂起到朗读完成、失败或被取消。
- `stop()`：停止当前朗读，并让等待中的调用以取消或明确结果结束。
- `isSpeaking` 继续保留，用于 UI 和状态判断。

实现上使用 Android `TextToSpeech.OnUtteranceProgressListener` 追踪 `onDone`、`onError` 和 `onStop`。

### AdaptiveAudioRecorder

将固定 4 秒录音替换为动态短录音：

- 最短录音时长：约 800ms，避免刚开始说话就被静音检测截断。
- 静音结束阈值：检测到约 900ms 连续低音量后结束录音。
- 最长录音时长：约 8s，避免无声环境或持续噪声导致无限等待。
- 无有效语音时返回“我没有听清，请再说一次。”，连续聆听模式下不反复播报这句。

V2 第一阶段可以继续使用 `MediaRecorder` 或改为 `AudioRecord`。如果需要基于音量做可靠静音检测，应优先改用 `AudioRecord`，因为它能读取 PCM 帧并计算能量；不引入流式 ASR。

### AssistantSessionManager

`AssistantSessionManager` 保持主编排职责，但语音细节下沉：

- 负责决定一次识别文本是否为停止连续聆听、确认高风险操作或普通请求。
- 负责屏幕采集、调用后端 AI、校验动作协议、风险确认和动作执行。
- 负责把状态变化反馈给浮窗和通知。
- 不直接 `delay` 等待 TTS，也不直接控制录音器细节。

## 状态与取消语义

V2 的浮窗点击和停止指令必须根据当前状态执行唯一动作：

| 当前状态 | 浮窗点击 | 停止类语音指令 |
| --- | --- | --- |
| `SpeakingPrompt` / `SpeakingResult` | 停止朗读，退出当前轮 | 下一轮识别到后停止连续聆听 |
| `Listening` | 停止录音，取消当前轮 | 本轮识别为停止指令时不调用 AI |
| `Transcribing` / `Thinking` | 取消当前请求并播报“已取消。” | 不适用，尚未进入下一轮识别 |
| `Acting` | 尽力取消后续动作，不回滚已执行动作 | 不适用 |
| `ContinuousIdle` | 关闭连续聆听 | 关闭连续聆听 |

取消高风险待确认操作沿用现有 `ConfirmationManager`，但确认提示必须通过 `speakAndAwait` 完成后才进入下一轮录音。

## 错误处理

- TTS 初始化失败：回退为不等待的空输出，但必须阻止录音和朗读并发；同时在可用时提示“语音输出不可用，请检查系统 TTS 设置。”
- 录音无权限：朗读“麦克风权限未开启，请先在应用首页开启。”
- 录音无有效语音：一次唤起模式朗读“我没有听清，请再说一次。”；连续聆听模式下静默进入下一轮，避免噪声环境反复打扰。
- ASR 超时或网络失败：保留 V1 的清晰语音提示，并等待提示播完后再进入下一轮。
- AI 请求或动作执行失败：保留现有错误提示，提示播完后再决定是否继续聆听。

## 人性化体验要求

- 连续聆听开启时只播报一次“连续聆听已开启”，不要每轮都说“请说”。
- 用户停止连续聆听时只播报一次“已停止聆听。”
- 录音开始前可使用极短提示音或短提示语；若使用提示语，必须等待提示结束后再录音。
- 用户说短命令时应尽快结束录音，不强迫等待固定 4 秒。
- 用户说长一点的自然句时，录音应允许到最长 8 秒，而不是截断在 4 秒。

## 测试标准

### 单元测试

- TTS 未完成时不会开始录音。
- 提示语播放完成后才进入录音。
- 动态录音在检测到有效语音后的静音尾段结束。
- 动态录音达到最长时长后强制结束。
- 连续聆听不会在结果播报完成前进入下一轮。
- 浮窗点击在朗读、录音、AI 请求中分别执行正确取消行为。
- 停止指令不会调用屏幕采集或 AI。
- 高风险确认提示播完前不会进入下一轮录音。

### 集成测试

- Fake TTS + Fake Recorder + Fake ASR 验证完整 turn 顺序。
- Mock AI Proxy 验证语音识别、屏幕采集、AI 响应、动作执行和结果播报按顺序完成。
- 网络失败、ASR 失败、权限关闭、无语音输入都不会导致连续聆听状态卡死。

### 人工验收

在本机安卓虚拟机上验证：

- 助手播报“当前页面...”时，下一轮不会把这句话识别成用户命令。
- 说“点击设置里的 WLAN”这类超过 4 秒的慢速语句不会被提前截断。
- 说“返回”这类短命令时，不需要等满 4 秒才处理。
- 连续聆听时说“停止聆听”后，不再录音、不再请求 AI。
- 浮窗在朗读中点击会停朗读，在录音中点击会取消录音，在请求中点击会取消当前请求。

## 明确不做

- 不做全天候热词唤醒。
- 不做隐藏后台录音。
- 不做流式 ASR 或实时对话。
- 不替换现有 SightSync Android App。
- 不新增或自行切换云端 AI provider/model。
- 不扩展 V2 手机控制动作白名单；手机控制能力另开 V2 执行标准和实现计划。
