# 安卓盲人 AI 无障碍助手 V1 纲领

## Summary

目标：做一个可用但克制的 Android V1：盲人用户通过浮窗/通知入口唤起语音助手，助手读取当前手机界面、用云端 AI 解释界面，并通过 `AccessibilityService` 执行基础操作。

已选方案：

- Android 原生 Kotlin + Jetpack Compose
- `minSdk 30`，优先 Android 11+
- 云端 AI API 通过后端代理接入，不在 App 内硬编码 key
- 主交互：浮窗/通知入口 + 语音输入 + TTS 反馈
- V1 只做基础手机操作，不做全天候语音唤醒、离线大模型、复杂自动化流程

## Phase 1: 项目骨架与权限入口

实现内容：

- 创建 Android App 工程：Kotlin、Compose、MVVM、Coroutines、Retrofit/OkHttp。
- 创建极简 AI Proxy 后端：提供 `POST /v1/assist`，负责调用云端 AI、保存 API key、校验结构化输出。
- App 首页只做三件事：开启无障碍服务、开启麦克风权限、开启悬浮窗权限。
- 增加隐私说明页，明确会上传当前屏幕文本/截图给 AI 服务处理。

关键组件：

- `MainActivity`
- `PermissionViewModel`
- `AssistantAccessibilityService`
- `OverlayController`
- `AiProxyClient`

验收标准：

- 用户安装后能进入引导页。
- 能跳转到系统无障碍设置并启用服务。
- 启用后能看到无障碍浮窗或常驻通知入口。

## Phase 2: 语音交互闭环

实现内容：

- 点击浮窗后开始语音识别。
- 识别完成后播放“正在查看当前屏幕”之类的 TTS 状态反馈。
- 支持中断：用户再次点击浮窗可停止朗读或取消当前请求。
- 所有结果都通过语音播报，不依赖视觉 UI。

关键组件：

- `SpeechInputController`
- `TtsOutputController`
- `AssistantSessionManager`

V1 指令示例：

- “这里有什么？”
- “读一下当前页面”
- “点击确定”
- “返回”
- “向下滚动”
- “输入：你好，我到了”

验收标准：

- 用户不看屏幕，也能完成一次语音提问和语音回答。
- 网络失败、识别失败、AI 超时都有清晰语音提示。

## Phase 3: 屏幕理解能力

实现内容：

- 从 `AccessibilityNodeInfo` 提取当前屏幕结构：文本、按钮、输入框、可点击项、可滚动容器、包名、窗口标题。
- Android 11+ 使用 `AccessibilityService.takeScreenshot()` 获取截图，作为 AI 辅助理解材料。
- 对密码框、验证码、银行卡号等敏感字段做本地脱敏。
- 先优先使用节点树；只有节点树不足时才附带截图，降低隐私风险和成本。

核心数据结构：

```kotlin
data class ScreenContext(
    val packageName: String,
    val activityName: String?,
    val nodes: List<ScreenNode>,
    val screenshotBase64: String?
)

data class ScreenNode(
    val nodeId: String,
    val text: String?,
    val contentDescription: String?,
    val role: String,
    val bounds: Rect,
    val clickable: Boolean,
    val editable: Boolean,
    val scrollable: Boolean
)
```

验收标准：

- 在系统设置、短信/聊天列表、浏览器网页等常见页面，能朗读主要内容。
- AI 回答中能准确说出当前页面的主要控件和可执行动作。

## Phase 4: AI 规划与动作协议

实现内容：

- App 将用户语音 + `ScreenContext` 发给后端代理。
- 后端要求 AI 只返回结构化 JSON，不允许返回任意脚本或自由执行命令。
- Android 端只执行白名单动作。

后端接口：

```http
POST /v1/assist
Authorization: Bearer <app-token>
Content-Type: application/json
```

请求：

```json
{
  "sessionId": "uuid",
  "locale": "zh-CN",
  "utterance": "点击确定",
  "screen": {
    "packageName": "com.android.settings",
    "activityName": "Settings",
    "nodes": [],
    "screenshotBase64": null
  }
}
```

响应：

```json
{
  "spoken": "我会点击“确定”。",
  "requiresConfirmation": false,
  "actions": [
    {
      "type": "CLICK_NODE",
      "nodeId": "node_12"
    }
  ]
}
```

允许的 V1 动作：

- `SPEAK`
- `CLICK_NODE`
- `SET_TEXT`
- `SCROLL_FORWARD`
- `SCROLL_BACKWARD`
- `GLOBAL_BACK`
- `GLOBAL_HOME`
- `OPEN_APP`

验收标准：

- “这里有什么”只朗读，不执行动作。
- “点击确定”能点击明确匹配的按钮。
- 如果 AI 无法确定目标，必须追问用户，而不是猜测执行。

## Phase 5: 安全执行层

实现内容：

- `ActionExecutor` 负责执行 AI 返回的动作。
- 点击前校验目标节点仍然存在、可点击、屏幕包名未变化。
- 涉及支付、删除、发送消息、提交表单等高风险动作时，强制二次语音确认。
- 如果页面变化导致节点失效，重新采集屏幕并提示用户重试。

关键组件：

- `ActionExecutor`
- `RiskClassifier`
- `NodeMatcher`
- `ConfirmationManager`

验收标准：

- 基础点击、返回、主页、滚动、输入文本可用。
- 高风险动作不会被一次语音直接执行。
- 错误执行失败时，有明确语音反馈。

## Phase 6: V1 测试与验收

单元测试：

- `ScreenContextCollector` 节点树提取。
- 敏感字段脱敏。
- AI 响应 JSON 解析与非法动作拒绝。
- `RiskClassifier` 高风险动作判断。
- `NodeMatcher` 文本匹配、模糊匹配、多结果处理。

集成测试：

- Mock AI Proxy，验证完整链路：语音指令 -> 屏幕采集 -> AI 响应 -> 动作执行。
- AccessibilityService 在测试页面中执行点击、输入、滚动。
- 网络超时、AI 返回非法 JSON、权限关闭等失败场景。

人工验收场景：

- 打开系统设置，说“这里有什么”。
- 说“点击 WLAN”或页面上的明确按钮。
- 在输入框页面说“输入：测试内容”。
- 在长页面说“向下滚动”。
- 在任意页面说“返回”。
- 网络断开时，助手能朗读失败原因。

## Assumptions

- V1 优先做中文普通话体验。
- 首版不追求覆盖所有 App，只保证系统设置、浏览器、聊天/短信类常见页面具备基础可用性。
- 不做全天候热词唤醒，避免常驻录音、功耗和隐私复杂度。
- 不在 Android App 内保存云端 AI API key，统一走后端代理。
- 若后续要支持 Android 8-10，需要单独规划无截图模式或 MediaProjection 方案。
